# LRT-T-0010 · Harden extension bridge timeouts and MV3 wakeup recovery

## Intent
Stop native-host/extension restart churn from surfacing as random `extension_disconnected` workflow failures. The local runtime should distinguish stale/unready bridge state from step-level timeouts, bound extension calls, and give the MV3 worker more chances to reconnect the native host without making the Reason app a prerequisite.

## Read this when

Read this for task scope, acceptance, evidence, or implementation history.


## Scope

In:
- defect fix
- regression protection
- supervisor readiness probe
- content-script message timeout
- navigation-caused eval response loss
- MV3 content-script wakeup
- workflow failure error normalization

Out:
- unrelated cleanup
- cloud actor ownership migration
- Reason app migration
- replacing the Chrome extension/native-host transport

## Symptom

- `rzn-browser run google search` failed with `Reason: extension_disconnected` even while the extension card only showed normal MV3 `service worker (Inactive)` state.
- The run could open a supervisor session and complete early steps, then hang for 40s on a browser step.
- After killing/restarting the Chrome-launched native host, the supervisor correctly stayed alive but Chrome did not immediately launch a replacement native-host process while the MV3 worker was asleep.

## Reproduction

1. Install the supervisor runtime and run `rzn-browser run google search --param search_query="rzn browser smoke" --param vertical=web`.
2. Observe the original failure on `execute_javascript` with `Native-host extension bridge timeout after 40000ms`.
3. After the eval fix, rerun and observe progress through `execute_javascript`; the next unbounded hang reproduced on `dismiss_popups`.
4. Kill the Chrome-launched native host and run `rzn-browser supervisor ensure-ready --json`.

Expected:
- Step-level hangs are bounded and reported as timeouts, not stale extension disconnects.
- `runtime.ensure_ready` proves extension responsiveness with an active ping.
- Normal content-script activity can wake the background worker and reconnect the native host after restart churn.

Observed:
- Bridge existence alone was treated as readiness.
- `sendMessageTopFrameWithRetry` accepted a timeout option but did not actually timeout `chrome.tabs.sendMessage`.
- `execute_javascript` could lose its reply when the script triggered navigation.
- `dismiss_popups` had no count/time cap.

## Constraints

- Do not make the Reason app a hidden runtime prerequisite.
- Do not replace the product path with Playwright/WebDriver.
- Native messaging remains Chrome-owned; CLI can start supervisor but cannot directly force Chrome to call `connectNative` unless the extension is awake.

## Escalate if

- Chrome keeps refusing to start the native host after an extension reload.
- `runtime.ensure_ready` reports `connected=true` but `responsive=false` after reload.
- The supervisor process exits during a bounded browser call.
