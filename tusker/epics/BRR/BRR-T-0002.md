# BRR-T-0002 · Reconnect native messaging on MV3 service-worker wake

## Agent capsule

- Essence: Reconnect native messaging on MV3 service-worker wake.

- Avoid: raw logs, full transcripts, generated indexes, and attachments unless doing evidence forensics.

## Intent

Reconnect the extension native messaging port whenever the MV3 service worker
wakes. Current reconnect calls are event-driven; if the worker wakes for a
reason that only runs top-level code/listener registration, `nativePort` can
remain null until a later alarm or inbound event. Top-level reconnect should be
idempotent and guarded by the existing in-flight flag.

## Acceptance contract

| # | Outcome | Proof required | Docs impact |
|---|---|---|---|
| 1 | Background startup/wake schedules an immediate `connectToNative()` when no native port exists and no connect is in flight. | Extension unit test or focused review plus build proves guarded top-level path. | `docs/features/bridge_reliability_hardening/README.md` task status. |
| 2 | Reconnect remains idempotent and does not create concurrent native ports. | Test or code evidence around `nativeConnectInFlight`/`nativePort` guard. | none |
| 3 | Existing event-driven reconnect paths still work. | Existing extension tests/build pass. | none |

## Symptom

- After MV3 eviction, the background global `nativePort` is gone. On wake,
  listeners are registered but native messaging may not reconnect until an
  alarm, startup, install, or inbound message path fires.

## Reproduction

1. Let the extension service worker become inactive.
2. Wake it through a path that does not call the existing reconnect handlers.
3. Observe `nativePort` remain null until the next reconnect event.

Expected:
- Top-level SW wake attempts native messaging reconnect immediately.

Observed:
- The service worker can be awake but disconnected from the native host.
