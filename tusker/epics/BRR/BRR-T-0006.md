# BRR-T-0006 · Add sub-30s bridge heartbeat and health beacon

## Agent capsule

- Essence: Add sub-30s bridge heartbeat and health beacon.
- Next action: define acceptance, do the smallest scoped change, and attach concise evidence.
- Read next: this note, then only the code/docs anchors named here.
- Avoid: raw logs, full transcripts, generated indexes, and attachments unless doing evidence forensics.

## Intent

Add a stronger liveness and recovery mechanism because cheaper recovery is now
proven insufficient under repeated workflow load. Native-host heartbeat frames
may keep the MV3 service worker active, but the observed failure is sharper:
after a step-level bridge timeout, the native-host process can remain alive and
registered while it no longer forwards extension replies. Heal can then fail
until the extension is manually reloaded or the native-host/native-port path is
explicitly restarted.

## Acceptance contract

| # | Outcome | Proof required | Docs impact |
|---|---|---|---|
| 1 | Zombie native-host bridge is detected: process/fd/socket alive but ping and step replies no longer return. | Regression test or harness proves supervisor classifies this separately from simple bridge down. | Feature scratchpad updated with field failure. |
| 2 | Recovery can restart/recreate the native-host/native-port path without manual extension reload. | Live smoke: step timeout -> heal/recovery -> readiness true -> next workflow proceeds. | Runtime docs mention recovery path. |
| 3 | If heartbeat is implemented, interval is bounded, low overhead, and disabled/guarded when native messaging is disconnected. | `cargo test -p rzn-native-host` plus code review of timer lifecycle. | Runtime docs mention background heartbeat behavior. |
| 4 | Health beacon records last successful extension ping/build signature and last failed ping/step timeout so diagnostics show beacon age and failure transition. | Supervisor tests cover fresh beacon, stale beacon, and zombie transition decisions. | Runtime docs/status docs if schema changes. |

## Symptom

- After 4-5 workflows, a browser step can time out with
  `Native-host extension bridge timeout after 40000ms`.
- Immediately after that, `runtime.ensure_ready` reports
  `cause=transport_timeout` / target `cause=zombie_native_host`,
  `native_host_bridge.connected=true`,
  `responsive=false`, and no loaded extension metadata because ping gets no
  response.
- Killing the native-host process leaves `bridge_down`; current heal does not
  make the extension respawn the native host within its wait window.

## Reproduction

1. Start current debug supervisor and verify readiness succeeds.
2. Run `rzn-browser run google search --param search_query="rzn bridge smoke" --param vertical=web`.
3. Observe `browser.session_open` succeed, then `s1 (navigate_to_url)` time out
   with `Native-host extension bridge timeout after 40000ms`.
4. Run `runtime.ensure_ready`.
5. Observe `cause=transport_timeout`, `connected=true`, `responsive=false`.
6. Kill the native-host process.
7. Run `rzn-browser heal --json`.

Expected:
- Heal/recovery restarts the native-host/native-port path and returns ready.

Observed:
- Heal remains `ready=false` and reports `bridge_down`; manual extension reload
  is still required.
