# BRR-T-0005 · Decouple readiness capability checks from bridge transport health

## Agent capsule

- Essence: Decouple readiness capability checks from bridge transport health.
- Next action: define acceptance, do the smallest scoped change, and attach concise evidence.
- Read next: this note, then only the code/docs anchors named here.
- Avoid: raw logs, full transcripts, generated indexes, and attachments unless doing evidence forensics.

## Intent

Separate "can talk to the extension" from "loaded extension supports this
capability." Today `content_keepalive_port` is used as a hard readiness gate;
that is useful for stale-bundle detection but can also make transport appear
broken when the actual issue is version/capability mismatch.

## Acceptance contract

| # | Outcome | Proof required | Docs impact |
|---|---|---|---|
| 1 | Readiness reports transport health independently from required capability health. | Rust tests cover `transport_ok=true, capability_ok=false`. | Runtime docs/status docs updated if schema changes. |
| 2 | Workflows can declare/require capabilities before dispatch, or the supervisor has an explicit policy for global gates. | Manifest/schema or supervisor policy test, depending on chosen design. | Workflow/runtime docs updated. |
| 3 | Stale bundle guidance is explicit and actionable. | CLI/status output includes loaded build/capability where available. | `docs/features/bridge_reliability_hardening/README.md` task status. |

## Symptom

- Old extension bundle returns a ping but lacks required capability, and the run
  fails like a bridge readiness problem.

## Reproduction

1. Load a pre-`content_keepalive_port` extension bundle.
2. Run readiness or a CLI workflow.

Expected:
- Transport is reported as alive, with a stale bundle/capability warning or a
  workflow-specific capability block.

Observed:
- Readiness hard-fails with a generic bridge readiness message.

## Scope

In:
- supervisor readiness/capability model
- workflow capability requirement policy if needed
- CLI guidance for stale bundles

Out:
- Removing capability checks entirely.
- Masking stale bundles for workflows that require the missing capability.

## Deliverables

- Decoupled readiness fields/policy.
- Tests for transport-ok/capability-missing.
- Updated docs if public JSON or workflow manifest semantics change.

## Constraints

- Do not let genuinely unsupported workflows run blindly.
- Keep the operator/user action clear: reload/upgrade extension when the loaded
  bundle is stale.

## Escalate if

  would balloon beyond this epic.
