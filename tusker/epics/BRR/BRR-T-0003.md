# BRR-T-0003 · Retire stale native-host bridge handles on probe timeout

## Agent capsule

- Essence: Retire stale native-host bridge handles on probe timeout.
- Next action: define acceptance, do the smallest scoped change, and attach concise evidence.
- Read next: this note, then only the code/docs anchors named here.
- Avoid: raw logs, full transcripts, generated indexes, and attachments unless doing evidence forensics.

## Intent

Stop preserving dead native-host bridge handles after readiness probe timeout.
The supervisor currently reports `connected=true` when its stdio fd remains
open, even if the extension side cannot answer. A timed-out active ping should
mark the bridge stale, drop or quarantine the cached handle, and force the next
call through reconnect/heal logic.

## Acceptance contract

| # | Outcome | Proof required | Docs impact |
|---|---|---|---|
| 1 | Probe timeout/`Ok(None)` retires or marks stale the cached native-host bridge. | `cargo test -p rzn-browser supervisor` covers timeout -> stale/removed bridge state. | `docs/features/bridge_reliability_hardening/README.md` task status. |
| 2 | Supervisor status distinguishes fd-open stale handle from live responsive bridge. | JSON/status test includes stale or responsive state. | Runtime docs if status schema changes. |
| 3 | A later call can reconnect instead of reusing the dead handle indefinitely. | Test or smoke proves post-timeout reconnect path is reachable. | none |

## Symptom

- `native_host_bridge.connected=true` but readiness ping times out. The fd can
  linger while Chrome/native-host/SW teardown is still in progress or wedged.

## Reproduction

1. Establish native-host bridge.
2. Kill/suspend the extension side or force a ping timeout while the host fd
   remains open.
3. Call readiness/status repeatedly.

Expected:
- First failed active probe marks the bridge stale and subsequent checks attempt
  reconnect/heal.

Observed:
- Supervisor can keep reporting a connected bridge that is not responsive.
