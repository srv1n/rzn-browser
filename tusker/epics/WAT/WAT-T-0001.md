# WAT-T-0001 · Consolidate X read workflows into x_open + add CLI write/download

## Intent

Re-validation of the X read workflows surfaced two problems while opening `https://x.com/addyosmani/status/2053231239721885918?s=12`:

1. The status URL was actually a longform X article. `x_open_post` returned only the t.co stub for `text`. `x_open_article` (when invoked manually with the derived `/article/<id>` URL) extracted the body but mis-picked title (`X`), author (`View keyboard shortcuts / i`), and joined paragraphs without separators.
2. There was no HTML→Markdown converter anywhere in the repo (only `format_markdown_results()` for search-result arrays in `crates/rzn_browser/src/result_formatter.rs:160`), and no way to dump the workflow result to a file or download referenced media + linked pages locally.

## Read this when

Read this for task scope, acceptance, evidence, or implementation history.


## Scope

In:
- Replace `x_open_post`, `x_open_article`, `x_thread` with one consolidated `x_open` workflow that auto-detects kind from URL + DOM, redirects `/status/<id>` → `/article/<id>` when the input is a longform-article launcher, scroll-collects same-author chains for threads, and converts the article body to markdown inline with a self-contained HTML→MD walker.
- Wire CLI `--output-file <path>` and `--download-dir <dir>` flags into the `rzn-browser run` surface (also accept `output_file`/`output_path`/`download_dir` workflow params). Runner functions (`run_*_workflow`) now return `Result<Option<Value>>` so the CLI can post-process. Downloads cover image_urls + video_urls + external_links and emit a `manifest.json`.
- Update workflow catalog cards (`resources/cards/social/x.json`, `x_v1.json`, `x_browser_profile_v1.json`), parse test, and READMEs to reflect the merge.

Out:
- Title-extraction polish: the X article surface still serves `<title>X</title>` and an `og:title` of `Addy Osmani on X: "Agent Harness Engineering" / X`. Current fallback chain prefers an in-article `h1` and falls through to og:title; on this page the in-article H1 was the X branding chip, so the visible H1 in markdown is `# X`. Body/markdown is intact — title is cosmetic.
- Mutating X workflows (like, reply, create, send-dm, reply-dm) untouched.
- Native engine step types unchanged (no new `write_file` step type — file writes happen CLI-side).
