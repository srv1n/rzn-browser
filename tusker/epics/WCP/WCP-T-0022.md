## Follow-up implementation — 2026-05-10

The first backend review was right: this was not ready for broad rollout with the runtime still drifting from the manifest. Follow-up implementation landed the minimum bridge so `x/open` is a real authoring pilot rather than contract-shaped JSON beside legacy runtime JSON.

Closed or reduced:
- `runtime.workflow_ref` / `runtime.workflow_path` are typed ManifestV2 fields. Strict catalog validation now errors when `steps[]` is empty without a runtime pointer, and it checks `result.output_selector.step_id` against the referenced legacy workflow.
- `ActionKindV2` now covers the known engine step types used by the browser runner instead of forcing common steps through `Custom`.
- The CLI runner discovers the owning ManifestV2 for a legacy workflow, then injects `workflow_id`, `workflow_version`, `system`, `capability`, and enforced declared `side_effects` into every `browser.execute_step` payload.
- Supervisor `RunResultV2.workflow_id` now inherits workflow identity from step params instead of falling back to `rzn.local.browser.execute_step`.
- CLI final stdout is wrapped as `RunResultV2`, and `--output-file` / `--download-dir` unwrap `RunResultV2.output` before writing/downloading.
- `workflow validate <manifest> --strict --json` now validates ManifestV2 through `rzn_contracts` directly and checks the runtime pointer; it no longer treats manifest files as legacy workflow help docs.

Remaining before broad rollout:
- Manifest `steps[]` is still not authoritative; the runtime bridge is explicit now, but the runner still executes legacy workflow JSON.
- Manifest `params` still duplicate legacy `required_variables` / `optional_variables`.
- CLI post-processing file writes/downloads are still outside supervisor policy observation.
- No `network_egress` / `external_read` side-effect class yet.
- Convert a second low-risk workflow (`pubmed/search` or `google/search`) before calling this pattern settled.

## Friction log (new ManifestV2 contract)

1. **Manifest `steps[]` is parallel to runtime `browser_automation.sequences[].steps[]`**. Runner ignores manifest steps, reads legacy file. Contract validates green even with `steps:[]`. Either add `runtime.workflow_path` pointer or make manifest steps authoritative.
2. **`ActionKindV2` only enumerates 11 kinds; engine ships 25 step types**. `dismiss_popups`, `wait_for_element`, `scroll_window_to`, `extract_structured_data`, `same_origin_request`, `assert_selector_state`, `infinite_scroll`, `request_user_intervention`, `take_screenshot`, `open_new_tab`, `close_current_tab` all collapse to `Custom { custom_kind }`. That makes `Custom` the norm, not the escape hatch. Either expand the enum or add a `custom_kind` registry inside `rzn_contracts`.
3. **`params` in ManifestV2 duplicates legacy `required_variables` / `optional_variables`** in `browser_automation.sequences[]`. Two sources of truth; runner uses the legacy one. Pick one.
4. **`runtime.actor`** has no `desktop` variant. OK if desktop folds into `supervisor`, but worth a doc note.
5. **`output_selector.step_id`** has no validator cross-check that the referenced step actually exists in `steps[]` (or in the runtime workflow). Silent drift if a step is renamed.
6. **CLI-side file writes are not policy-tracked.** `--output-file` and `--download-dir` happen in the CLI post-processor (`crates/rzn_browser/src/main.rs:1951 process_workflow_output`). `observed_side_effects()` only inspects supervisor calls — never sees these writes. Manifest-declared `file_write` + `download` are advisory for these paths.
7. **`local_tool_workflow_id`** (`supervisor.rs:761`) defaults to `rzn.local.<tool_name>` because `browser.execute_step` payloads do not carry `workflow_id`. Most `RunResultV2` emissions therefore have a meaningless workflow_id. Threading the active workflow id through every step call would fix this.
8. **No `network_egress` / `external_read` side-effect class.** The downloader hits arbitrary external HTTP endpoints; `download` covers the file write but elides the network surface.
9. **`RetryPolicyV1::default().max_attempts = 1`** — ambiguous. Doc the semantics or rename to `max_total_attempts` / `max_retries`.
10. **`workflow validate --strict <manifest>` requires the workflow be installed in the catalog** ("Strict mode requires the workflow to be installed in the catalog"). For repo-local manifests being authored, that's a chicken-and-egg ergonomic. `validate-catalog --strict` works; per-file strict does not.
11. **`run_result` field is built supervisor-side but never reaches CLI stdout.** `extract_payload_for_output` (`native_runner.rs:2734`) reads `response.result` / `response.data`, never `response.run_result`. Only MCP consumers see the `RunResultV2`. CLI users get the legacy step payload only.
12. **Declared `side_effects` not auto-enforced.** `enforce_manifest_side_effect_policy` (`supervisor.rs:980`) only fires when params include `side_effect_policy.enforce: true`. The CLI runner never injects this. Declared side effects are advisory at runtime today.

## Recommendation

Pilot is healthy as a contract-authoring exercise — strict catalog flips green, capability listing works, runtime still passes. Do **not** promote broadly until items 1, 2, 3, 11, and 12 above are addressed. Best next step: thread the manifest's declared `side_effects` + active `workflow_id` through `browser.execute_step` automatically, then convert one more low-risk read workflow (`pubmed/search` or `google/search`) the same way.
