## Bug list

### P0 — silent failure / silent author trap

#### B1. `runScriptingEval` swallows eval errors as `success: true, result: null`
**Path:** `extension/src/background.ts:2013` (`result: result?.result ?? null`).
**Symptom:** When the injected function throws (`ReferenceError`, `TypeError`, etc.), `chrome.scripting.executeScript` returns `[{ result: undefined }]`. The runner maps that to `success: true, result: null`. Workflow status shows `succeeded`. Output file holds `{execution_backend, result: null, success: true, world}`. Author has zero signal.
**Repro:** Any `execute_javascript` step where the script reads `arg3` (or any undeclared identifier under `"use strict"`).
**Fix sketch:** Inspect `result?.error` (chrome.scripting populates it on uncaught throws), or wrap the injected `func` in a top-level `try/catch` and return `{ __rzn_eval_error: { message, stack } }`. Surface as `success: false`.

#### B2. Eval wrappers cap positional `args` at `arg0..arg2` — `arg3+` ReferenceErrors silently
**Paths:**
- `extension/src/contentScript.ts:598-600`
- `extension/src/pageBridge.ts:319-321`
- `extension/src/background.ts:1723-1725`
- `extension/src/background.ts:2078-2080`

All four wrappers declare `const arg0 = __args[0]; const arg1 = __args[1]; const arg2 = __args[2];` and stop. Workflow scripts referencing `arg3+` ReferenceError under `"use strict"`. Combined with B1, the failure is invisible.
**Hit on 2026-05-10:** `workflows/chatgpt/chatgpt_read.json` (originally referenced `arg3` for `download_attachments`). Worked around by moving extras into `window.__rzn_params`. Other workflow authors will hit this same trap — there is no compile-time check.
**Fix sketch:** Either declare `arg0..arg9` uniformly across all four wrappers (cheap), or deprecate the `args` array path entirely and force everything through `__rzn_params` (cleaner; runner already injects every workflow param into `params` at `crates/rzn_browser/src/native_runner.rs:2731 inject_script_params`).

### P1 — contract drift / runtime drift

#### B3. Manifest `steps[]` parallel to legacy `browser_automation.sequences[].steps[]`
**Path:** `crates/rzn_browser/src/native_runner.rs:2475` (`manifest_step_to_legacy_step` round-trip), `extract_steps` at `:2788` reads only legacy.
**Symptom:** Manifest contract validates green even with `steps:[]`. Runner ignores manifest's authoritative steps. Today `WorkflowManifestV2 → legacy adapter` works but is two parallel sources of truth.
**Fix sketch:** Have runner consume `manifest.steps[]` directly. Drop the adapter once one workflow runs without legacy shape.

#### B4. Manifest `params` duplicates legacy `required_variables` / `optional_variables`
**Path:** `crates/rzn_browser/src/native_runner.rs:2460-2473` (adapter regenerates legacy var lists from manifest params).
**Symptom:** Two sources of truth. Runner uses the legacy one. Drift waiting to happen.
**Fix sketch:** Remove the legacy var lists once `manifest.params` is the sole input contract.

#### B5. `ActionKindV2` only enumerates 11 of 25 engine step types
**Path:** `crates/rzn_contracts/src/v2.rs` (or wherever `ActionKindV2` lives).
**Symptom:** `dismiss_popups`, `wait_for_element`, `scroll_window_to`, `extract_structured_data`, `same_origin_request`, `assert_selector_state`, `infinite_scroll`, `request_user_intervention`, `take_screenshot`, `open_new_tab`, `close_current_tab`, etc. all collapse to `Custom { custom_kind }`. `Custom` becomes the norm, not the escape hatch.
**Fix sketch:** Expand the enum to cover all engine kinds, OR add a typed `custom_kind` registry in `rzn_contracts`.

#### B6. `output_selector.step_id` not cross-checked against `steps[]`
**Path:** `crates/rzn_contracts/src/v2.rs` validator.
**Symptom:** Renaming a step silently breaks selection. Strict validate doesn't notice.
**Hit on 2026-05-10:** `workflows/chatgpt/chatgpt_send.json` selector points at `s11` but s11 only emits session metadata (`{chat_id, temporary_chat_active, title, url}`) — not the advertised `{message_sent, model_selected, effort_selected, tool_selected}`. Author error went undetected.
**Fix sketch:** Validator: assert `step_id` ∈ `steps[].id` (or `runtime.workflow_path` step ids).

#### B7. CLI-side file writes/downloads not policy-tracked
**Path:** `crates/rzn_browser/src/main.rs:1951 process_workflow_output`.
**Symptom:** `--output-file` and `--download-dir` happen post-supervisor. `observed_side_effects()` only inspects supervisor calls. Manifest-declared `file_write` + `download` are advisory for these paths.
**Fix sketch:** Funnel post-processor through supervisor, OR have post-processor emit synthetic side-effect events into the policy ledger.

#### B8. `RunResultV2` never reaches CLI stdout
**Path:** `crates/rzn_browser/src/native_runner.rs:2734 extract_payload_for_output` (reads `response.result` / `response.data`, never `response.run_result`).
**Symptom:** Only MCP consumers see `RunResultV2`. CLI users get the legacy step payload only. CLI _does_ now wrap output as `RunResultV2` envelope at the very end — but the `run_result` field built supervisor-side is dropped on the floor.
**Fix sketch:** Have `extract_payload_for_output` prefer `response.run_result` when present.

#### B9. Declared `side_effects` not auto-enforced
**Path:** `crates/rzn_browser/src/supervisor.rs:980 enforce_manifest_side_effect_policy`.
**Symptom:** Only fires when params include `side_effect_policy.enforce: true`. CLI runner never injects this. Declared side effects are advisory at runtime today.
**Fix sketch:** Default `enforce: true`. Allow `--unsafe` opt-out for debugging.

#### B10. No `network_egress` / `external_read` side-effect class
**Path:** `crates/rzn_contracts/src/v2.rs SideEffectClass`.
**Symptom:** The downloader hits arbitrary external HTTP endpoints (`x.com`, `pbs.twimg.com`, `gitconnected.com`, etc.). `download` covers the file write but elides the network surface.
**Fix sketch:** Add `network_egress` (any outbound HTTP/HTTPS) + `external_read` (read-only fetches). Update X + ChatGPT manifests to declare both where applicable.

#### B11. `RetryPolicyV1::default().max_attempts = 1` ambiguous
**Path:** `crates/rzn_contracts/src/v2.rs RetryPolicyV1`.
**Symptom:** Does `max_attempts: 1` mean "1 total attempt" or "1 retry on top of the original"? Field name doesn't disambiguate.
**Fix sketch:** Rename to `max_total_attempts` (preferred) or `max_retries`. Add doc comment + test.

#### B12. `workflow validate <manifest> --strict` requires catalog install
**Path:** `crates/rzn_browser/src/main.rs` (validate command path).
**Symptom:** Per-file strict validate errors with "Strict mode requires the workflow to be installed in the catalog" for repo-local manifests being authored. Chicken-and-egg ergonomic. `validate-catalog --strict` works; per-file strict does not for unsalted files.
**Fix sketch:** Per-file strict should validate the manifest standalone (no catalog lookup); only `validate-catalog --strict` should require installation.

### P2 — observability / nice-to-have

#### B13. `local_tool_workflow_id` defaults to `rzn.local.<tool_name>`
**Path:** `crates/rzn_browser/src/supervisor.rs:761`.
**Symptom:** Most `RunResultV2` emissions have a meaningless `workflow_id` because `browser.execute_step` payloads don't carry the active manifest id. Threading the active workflow id through every step call would fix it.
**Fix sketch:** CLI runner already discovers manifest id (verified — `chatgpt/read`, `chatgpt/send`, `x/open` show correct ids in the v2 envelope). Apply same threading to all `execute_step` paths in `supervisor.rs`.

#### B14. `chatgpt/send` returns sparse output
**Path:** `workflows/chatgpt/chatgpt_send.json` step `s11`.
**Symptom:** Manifest help advertises `{ chat_id, url, model_selected, effort_selected, tool_selected, message_sent }`. Actual return is `{ chat_id, temporary_chat_active, title, url }`. Send works (verified 2026-05-10 — message landed, assistant replied), but the output payload is misleading.
**Fix sketch:** Either have `s11` aggregate prior step outputs into the final selector, or change `output_selector.step_id` to a step that already aggregates. (This is a workflow-side fix, not engine-side, but is contract-adjacent — listing here for visibility.)
