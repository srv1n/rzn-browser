# WCP-T-0027 · Harden action and side-effect contract taxonomy

## Intent

Make the contract vocabulary match the real engine vocabulary so workflow
authors do not live in `Custom` and policy does not miss network/external-read
behavior.

## Read this when

Read this for task scope, acceptance, evidence, or implementation history.


## Scope

In:
- Expand `ActionKindV2` to cover every first-class engine step type used by production workflows.
- Decide whether `Custom` remains a rare escape hatch or becomes a registered extension point.
- Add side-effect classes for external/network reads where current `download`/`read_only` is too vague.
- Update migration script and production workflows to use the expanded taxonomy.
- Clarify retry attempt semantics.

Out:
- Runtime side-effect enforcement mechanics; that belongs in WCP-T-0028.
- Full workflow smoke; that belongs in WCP-T-0030.

## Acceptance

| # | Outcome | Proof required | Docs impact |
|---|---|---|---|
| 1 | First-class engine step types are represented in `ActionKindV2` | Contract tests cover every engine step type observed in production workflows | Authoring reference updated |
| 2 | Production workflows do not use `Custom` for known built-in actions | grep/jq report shows no known engine kind hidden in `custom_kind` | none |
| 3 | Network/external-read side effects are expressible | `SideEffectClass` includes explicit class(es), tests pass | Policy docs updated |
| 4 | Retry semantics are unambiguous | Field rename or doc/test explains total attempts vs retries | Contract docs updated |

## Constraints

- Avoid breaking already migrated manifests without a scripted migration.

## Escalate if

- The engine has dynamic plugin step kinds that cannot be enumerated cleanly.
- A side-effect class would overlap confusingly with an existing class.
