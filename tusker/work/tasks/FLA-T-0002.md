# FLA-T-0002 · Extract shared workflow run loop from native_runner

## Intent

Extract the workflow step-execution loop out of the CLI-only `native_runner.rs` into a shared module that both the CLI and the supervisor's fleet poll loop (FLA-T-0003) can call. The CLI's observable behavior must not change — this is a refactor with a hard regression bar, and it is the riskiest task in the epic.

## Context

- Today the run loop lives in `crates/rzn_browser/src/native_runner.rs` (~2,465 lines): manifest/legacy loading (`load_workflow_for_run` at `native_runner.rs:536-556`), param normalization (`normalize_manifest_params` `:616-655`), the per-step execution loop with transient retry (`:232-317`, `is_transient_step_error` `:43-54`, external-write no-retry guard `:27-41`), stop_workflow detection (`response_stop_reason` `:1427-1452`), output selection (`:1183-1233`) and `RunResultV2` assembly (`build_cli_run_result` `:1162-1181`).
- The loop currently drives steps by JSON-RPC over the supervisor local socket (`browser.execute_step` etc.) and prints `[OK]/[ERR]/[STOP]` progress lines directly.
- The supervisor will need to run the same loop in-process (it IS the socket server — it must not dial itself; see the transport trait below, whose in-process impl lands in FLA-T-0003).

## Contract

New module `crates/rzn_browser/src/workflow_runner/` (mod.rs + submodules as needed):

```rust
/// How the runner talks to the browser session layer.
#[async_trait]
pub trait StepTransport: Send + Sync {
    async fn call(&self, method: &str, params: serde_json::Value, timeout_ms: u64)
        -> Result<serde_json::Value, TransportError>;
}

/// Progress sink so the runner never prints directly.
pub trait RunEventSink: Send + Sync {
    fn on_step_start(&self, idx: usize, step_id: &str);
    fn on_step_finish(&self, idx: usize, step_id: &str, ok: bool, note: Option<&str>);
    fn on_stop(&self, reason: &str);
}

pub struct RunOptions { pub run_id: String, pub params: HashMap<String, String>,
    pub deadline: Option<Duration>, pub session: SessionSpec, /* browser target, existing-session reqs */ }

pub async fn execute_workflow(
    transport: &dyn StepTransport,
    sink: &dyn RunEventSink,
    workflow: LoadedWorkflow,        // parsed manifest-v2 or legacy, moved from native_runner
    opts: RunOptions,
) -> rzn_contracts::v2::RunResultV2
```

Moves (not copies) into `workflow_runner`: workflow loading/parsing for both formats, param normalization/injection, the step loop with transient retry + external-write guard + per-step watchdog, stop_workflow handling, output selection, RunResultV2 assembly. `native_runner.rs` keeps: CLI argument handling, the socket client (implements `StepTransport` over the existing local-socket JSON-RPC client), a `CliEventSink` that prints exactly the current `[OK]/[ERR]/[STOP]` lines, and the CLI-side post-processing (`--download-dir` attachment collection, `--output-file`) which stays where it is in `main.rs`.

Ordering rule for the refactor: keep function bodies byte-identical where possible — move code, do not rewrite logic. Where printing was interleaved with logic, replace with sink calls carrying the same strings.

## File ownership

- OWN: `crates/rzn_browser/src/workflow_runner/**` (new), `crates/rzn_browser/src/native_runner.rs` (shrinks), the `mod workflow_runner;` line in the crate root.
- OFF-LIMITS: `supervisor.rs`, `supervisor_cloud.rs`, `cloud.rs`, `main.rs` beyond the mod line if required, extension/**, all other crates.

## Constraints

- Zero behavior change for `rzn-browser run`: same stdout lines, same exit codes, same RunResultV2 shape, same retry/timeout constants (`DEFAULT_REQUEST_TIMEOUT_MS`, `DEFAULT_NATIVE_STEP_RPC_GRACE_MS`, watchdog).
- Unit-test the extracted loop with a `MockTransport` (scripted step responses): happy path, transient-error retry, external-write no-retry, stop_workflow early exit, per-step timeout, output selector.
- No AI attribution anywhere.

## Dependencies

- None (parallel-safe with FLA-T-0001; different files).
