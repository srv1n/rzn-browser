# SEC-T-0007 · Unify supervisor frame protocol in rzn_core and reconnect on broker stream errors

## Intent

The supervisor wire protocol (length-prefixed JSON frames over a local socket) is implemented twice with different size limits and different timeout semantics. The divergence already produces two real bugs: a 10MB-vs-16MB cap mismatch that errors on legal frames, and a cancellation-unsafe read that permanently desyncs the stream after any timeout. Move framing into `rzn_core` and make the client reconnect on stream errors.

## Context (read before coding)

Audit anchors (2026-06-11, re-verify):

1. `crates/rzn_plan/src/broker_client.rs:35` — `MAX_MESSAGE_SIZE` = 10MB (client read limit). `crates/rzn_browser/src/supervisor.rs:36` — `MAX_FRAME_SIZE` = 16MB (server write limit). A legal 12MB DOM/page-source frame from the supervisor kills the client read **after** the length prefix is consumed.
2. `broker_client.rs:349-355, 380-385` — `tokio::time::timeout(..., read_frame(...))`: when the timeout fires, `read_exact` is cancelled mid-frame, partial bytes are already consumed, and late responses stay buffered. The shared `supervisor_client` connection is only dropped on explicit disconnect (~line 1835), so after one 45s timeout every subsequent call misparses length prefixes — permanent desync until process restart.
3. Duplicated implementations: `send_frame`/`read_frame`/handshake/`spawn_supervisor` exist in both `broker_client.rs` (~2296, ~2524) and `supervisor.rs` (~4346, ~4638) with different semantics.

Background: cancellation-safety — `tokio::io::AsyncReadExt::read_exact` is NOT cancellation-safe; wrapping it in `timeout()` and continuing to use the same stream is always a bug. Correct patterns: (a) dedicated owned reader task that loops `read_frame` and forwards complete frames over an mpsc channel, with timeouts applied to the channel recv; or (b) treat any timeout/error as fatal for the connection and reconnect.

## Implementation plan

1. **New module `rzn_core::framing`** (or extend an existing transport module if one fits): `pub const MAX_FRAME_SIZE: usize`, `pub async fn write_frame(w, bytes)`, `pub async fn read_frame(r) -> Result<Vec<u8>>` with the length-prefix format exactly matching today's wire format (verify byte order/width from both existing impls before writing — they must agree today since they interoperate).
2. Pick ONE limit (16MB, the supervisor's) and use it on both sides. Document it on the const.
3. Port `supervisor.rs` and `broker_client.rs` to the shared module; delete the local copies. Keep the handshake logic where it lives if it differs per side, but framing primitives must be shared.
4. **Reconnect-on-error in broker_client**: restructure reads into an owned reader task (pattern (a) above). Any read error/timeout ⇒ mark connection dead, drop it, and lazily reconnect on next call. In-flight calls get a clear "connection reset" error instead of hanging or desyncing.
5. **Tests**: round-trip frame encode/decode; oversize frame rejected identically on both sides; desync regression test — open an in-memory duplex stream, time out one call mid-frame, then assert the next call either succeeds after reconnect or fails with "reset", never misparses. Run the existing supervisor/broker integration tests.

- Protocol/schema changes; wire format stays byte-identical.
