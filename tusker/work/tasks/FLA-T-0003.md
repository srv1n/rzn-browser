# FLA-T-0003 · Supervisor fleet poll loop with journal and result post

## Intent

Give the supervisor a fleet mode: a jittered poll loop that heartbeats to the backend, claims jobs, journals them to disk before execution, runs them through the shared workflow runner in-process, and posts results with crash-safe retry. This replaces the WebSocket actor (`supervisor_cloud.rs`) as the cloud attachment for the fleet design.

## Context

- Design: simple poll every 30–60s (client-side jitter), poll = heartbeat = claim; device token auth; `device_status` in the poll response tells a revoked/dormant device to stop; single_delivery jobs must never execute twice (journal before execute); on restart, accepted-but-unfinished jobs are reported `aborted` and unposted results are re-posted.
- Wire types: `rzn_contracts::fleet_v1` (FLA-T-0001). Runner: `workflow_runner::execute_workflow` + `StepTransport` (FLA-T-0002).
- The supervisor is the local-socket JSON-RPC server (`supervisor.rs`, method table at `supervisor.rs:1657-1691`); the fleet loop lives inside the same process, so its `StepTransport` impl calls `SupervisorState::dispatch(method, params)` directly (same funnel the cloud path already uses via `dispatch_cloud_command_to_extension`, `supervisor.rs:4437-4449`) — it must NOT dial its own socket.
- Existing reusable pieces: reqwest is already a dep (`cloud.rs` CLOUD_HTTP_CLIENT `cloud.rs:39-50`); health/readiness state exists (readiness causes like `bridge_down`/`zombie_native_host`, `supervisor.rs:76-82`); config persistence precedent `LocalCloudActorConfig` (`cloud.rs:258-271`) and secure file helpers in `rzn_core::secure_files`.
- Fleet config file (written by FLA-T-0005): `fleet_config.json` in the runtime dir, shape `{ "version": "rzn.fleet.device_config.v1", "server_url", "device_id", "device_token", "tenant_id", "poll_interval_seconds"? }`.

## Contract

New `crates/rzn_browser/src/supervisor_fleet.rs`:

1. **Startup hook**: in `supervisor serve` startup, if `fleet_config.json` exists and parses, spawn `run_fleet_loop(state, config)`. Absent config = no fleet behavior at all (local-only supervisors unchanged). Log one info line on start.
2. **Poll loop**: sleep `poll_interval_seconds` (server value from last response, else config, else 45) ± 33% jitter. Build `DeviceHealthV1` from supervisor readiness state. POST `{server}/v1/fleet/poll` with bearer token, body `FleetPollRequestV1 { health, active_job_ids, max_jobs: 1 }`. Network errors: log at debug, exponential backoff capped at 5 min, resume normal cadence on success. HTTP 403 with code `device_revoked`/`device_dormant`: log warn, STOP polling, set a `fleet_status` field readable via `cloud.status`-style local RPC (`fleet.status` method) so the CLI can display it.
3. **Journal** (`fleet_journal.jsonl` in runtime dir, append-only, fsync after write): entries `{job_id, state: accepted|running|finished|posted, terminal_status?, ts_ms, workflow_id, workflow_hash, single_delivery}`. Write `accepted` BEFORE any execution work. Compact on startup: rewrite keeping only non-`posted` entries.
4. **Execution**: one job at a time (v1). Resolve the manifest via the workflow cache (FLA-T-0004 `WorkflowCache::get(workflow_id, hash)`); journal `running`; run `execute_workflow` with an in-process transport, a sink that records step progress into the journal-adjacent in-memory state, params from assignment, deadline from `execution_deadline_seconds`; journal `finished` + terminal status.
5. **Cancellation**: poll response `cancellations` containing the running job's id → trigger a cooperative cancel (CancellationToken checked between steps by the runner sink wrapper; in-flight step completes, subsequent steps skipped); result status `cancelled`.
6. **Result post**: POST `/v1/fleet/jobs/:id/result` with `FleetResultPostV1`; on success journal `posted`. On failure keep retrying with backoff (and at every poll tick) until posted — journal survives restarts. `deduped: true` ack counts as posted.
7. **Startup reconciliation**: journal entries at `accepted`/`running` with no `finished` → construct a `FleetResultPostV1 { status: Aborted, error: "supervisor restarted mid-run", run_result: minimal RunResultV2 }` and post it. Entries `finished` but not `posted` → re-post the persisted result (persist the serialized result JSON next to the journal as `fleet_results/<job_id>.json` at `finished` time, delete after `posted`).
8. **Dedupe**: before executing an assignment, check journal — if job_id already `finished`/`posted`, skip execution and (re)post the stored result.

Unit tests with a mock HTTP server (e.g. spawn a tiny axum/hyper listener in-test — hyper is already in the tree via reqwest; or a `FleetApi` trait with a mock impl if that is cleaner): poll cadence decisions, claim→journal→execute(mock runner)→post happy path, crash-recovery reconciliation from a synthetic journal, revoked stop, cancellation path, result re-post after failed post.

## File ownership

- OWN: `crates/rzn_browser/src/supervisor_fleet.rs` (new), `fleet.status` RPC method registration + the startup spawn hook in `supervisor.rs` (keep to a handful of lines at clearly-marked points), in-process `StepTransport` impl (may live in `supervisor_fleet.rs`).
- OFF-LIMITS: `native_runner.rs`, `workflow_runner/**` (consume only), `cloud.rs`, `supervisor_cloud.rs` (leave the old actor untouched and dormant — removal is a separate cleanup once fleet is proven), extension/**.

## Constraints

- The fleet loop must never block the supervisor's local-socket serving (spawn on its own task; shared state via the existing Arc/Mutex patterns).
- Token handling: never log the device token; redact to first 8 chars of its hash if needed.
- All timestamps epoch ms; job execution uses a tokio timeout at `execution_deadline_seconds` producing `timed_out`.
- No AI attribution anywhere.
