## Intent

Prove the two repos work together: a scripted smoke test that starts the real backend (with `browser-fleet`), enrolls this machine's supervisor as a device, submits a job through the operator API, and verifies the laptop picks it up, executes it, and posts a result the operator can read. This is the epic's cross-repo gate and the demo of the whole design.

## Context

- Backend repo: `/Users/sarav/Downloads/side/rzn/backend` (sibling checkout), run with `cargo run --features browser-fleet` on a random high port with a temp sqlite file (`DATABASE_URL=sqlite:<tmp>/fleet-smoke.db`, `ENVIRONMENT=development`, `MIGRATE_ON_BOOT=true`). Dev environment allows the dev schema bootstrap; operator auth for the script comes from the backend's existing auth surface â€” create a user/session via the auth API (signup/login endpoints in `src/controllers/auth.rs`) or a tenant API key; DISCOVER the cheapest working path at implementation time and document it in the script header. Do not add any auth bypass to the backend.
- Two execution tiers, because a browser-driving run needs Chrome + the extension + native host installed (the Playwright e2e harness precedent at `extension/tests/e2e/native_host_smoke.spec.ts:57-162` shows supervisor + native host bootstrap):
  - Tier 1 (default, CI-safe): NO browser. Uses a headless-safe trivial workflow (a manifest whose single step is a `static_command`/no-op the supervisor can satisfy without the extension â€” check `browser.static_command` in the supervisor method table `supervisor.rs:2460-2469`; if nothing is truly extension-free, the script instead verifies claim + journal + an `aborted`/`failed` result posts correctly, which still proves the full transport loop).
  - Tier 2 (`--with-browser`): operator machine with Chrome running; uses `workflows/google/google_search.json` (published to the catalog by the script) and asserts the result payload contains search output.
- The script must leave no residue: temp dirs for backend db + a scratch runtime dir for an isolated supervisor instance (`RZN_RUNTIME_DIR`-style override â€” see `rzn_core::runtime_paths` for the env override the supervisor honors; use a dedicated dir so the developer's real supervisor/fleet config is untouched), kill spawned processes on exit (trap).

## Contract

New `scripts/fleet_smoke.sh` (bash, `set -euo pipefail`, ~200 lines max, heavily commented) with steps:
1. Build both sides (`cargo build --release -p rzn-browser -p rzn-native-host` here; `cargo build --features browser-fleet` in the backend path, overridable via `BACKEND_DIR` env).
2. Boot backend on `PORT=$((RANDOM+20000))`, wait for `/livez`.
3. Obtain an operator credential (documented discovery per Context), mint an enrollment code via `POST /v1/fleet/enrollment-codes`.
4. Start an isolated supervisor (`rzn-browser supervisor serve` with scratch runtime dir), run `rzn-browser fleet enroll --server http://127.0.0.1:$PORT --code $CODE --name smoke-device`.
5. Publish the tier-appropriate workflow manifest via `POST /v1/fleet/workflows`.
6. Submit a job (`POST /v1/fleet/jobs`, target `any`, 120s deadline), then poll `GET /v1/fleet/jobs/:id` up to 3 minutes until terminal.
7. Assert terminal status (tier 1: the documented expected status; tier 2: `succeeded` with non-empty output), pretty-print the result, exit 0/1 accordingly.
8. Also assert `GET /v1/fleet/devices` shows the device with a recent `last_seen_at` (heartbeat proof).

Plus a `README` section: `docs/FLEET_SMOKE.md` describing prerequisites, both tiers, and expected output.
