# SEC-T-0014 · LLM client and supervisor correctness batch: gemini key header, parts handling, backoff reset, restart policy

## Intent

Batch of verified medium-severity correctness bugs across the LLM clients and the supervisor, none big enough for its own task but each user-visible: API key in URLs, dropped parallel tool calls, panic paths, availability flaps, and unbounded history growth. Fix the checklist; each item is independent.

## Context + checklist (each anchor from the 2026-06-11 audit — re-verify before editing)

LLM clients (`crates/rzn_plan/src`):
1. `gemini_client.rs:347-350` — API key sent as `?key=` query param → leaks into proxies/logs. Fix: `x-goog-api-key` header.
2. `gemini_client.rs:221-266` — `convert_response` reads only `parts[0]`; parallel function calls and mixed text+call responses are silently dropped. Fix: iterate all parts, collect every function call + concatenate text parts.
3. `llm.rs:970-978` — `new_simple` panics via `unwrap_or_else(|| panic!(...))` on a bad API key. Fix: return `Result`.
4. `openai_client.rs` + `gemini_client.rs` `simple_chat` — `unwrap_or("")` converts refusals/missing content into empty-string success. Fix: missing content ⇒ `Err` with the finish/refusal reason.
5. `telemetry.rs:624` — `model_rates.get("gpt-4o").unwrap()` — fallback rate lookup can panic if the map changes. Fix: graceful default + log.
6. `llm_autonomous.rs:1246, 4742` — `conversation_history` is push-only across a ≤30-step loop with full DOM state per turn and a full clone per call → O(n²) token cost, eventual context overflow. Fix: prune — keep system + last N turns (N configurable, default ~8) + a compact summary of dropped turns; clamp DOM-state snippets to a max length.

Supervisor / transport (`crates/rzn_browser/src`, `crates/rzn_plan/src`):
7. `supervisor.rs:2921-2938` — ANY JSON-RPC error from the bridge triggers `request_native_bridge_restart`; benign command-level errors restart the native host (availability flap). Fix: restart only on transport-class failures (connection drop, handshake, timeout storm); command errors propagate to the caller.
8. `supervisor_cloud.rs:220, 327-336` — `reconnect_backoff` never resets after a successful connection; after a long healthy session, the next drop waits the previously-accumulated max. Fix: reset to base on successful connect (after a short stability window, e.g. 30s connected).
9. `supervisor.rs:2770-2776, 2880` — caller-supplied `timeout_ms` uncapped (u64::MAX accepted) and `deadline_at_ms` fixed at insert while the real wait starts after reconnect/resolution. Fix: clamp timeout to a sane max (10 min) and compute the deadline when the wait actually starts.
10. `broker_client.rs:595, 611, 630, 648` — `.to_fs_name()/.to_ns_name().unwrap()` on env-supplied `RZN_SOCK_PATH` — panic on invalid input. Fix: propagate as connect error. (If SEC-T-0007 restructured this file, rebase onto it — it has priority.)

SDK (`crates/rzn_sdk/src`):
11. `session.rs:188, 224-228` — `encoded_id` = `elem_{idx}` renumbered every snapshot; a held ID silently resolves to a different element after re-snapshot. Fix: include a snapshot generation in the ID (`elem_{gen}_{idx}`) and error on stale-generation lookups.
12. `tools.rs:71` + `session.rs:105` — `snapshot_max_elements` accepted in config but never applied. Fix: honor it in `snapshot()`.

## Implementation plan

Work the checklist in file order (one commit per numbered item or small group; cite the item number in the commit subject). Items 7-10 touch files SEC-T-0007 also touches — schedule this task after SEC-T-0007 merges. For every item add the smallest unit test that fails before the fix (the panic items can use `#[should_panic]` removal-style tests: assert `Err` instead).
