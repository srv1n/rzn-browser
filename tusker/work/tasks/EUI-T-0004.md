# EUI-T-0004 · Extension UI shell: RPC client, popup, badge

## Intent

Rebuild the extension's user-facing surface as a thin client over the supervisor RPCs: a typed RPC client, a new popup (health strip / now-running / master pause / recent runs), icon badge states, and the tabbed dashboard page scaffold that EUI-T-0005..0008 fill in. The existing popup is acknowledged placeholder code — replace it, don't refactor it.

## Context

- Extension is vanilla TS + Vite + zod (no UI framework), MV3. Existing files: `extension/src/popup.ts` (~17.5K, placeholder), `popup.css`, `extension/popup.html`, `vite.config.popup.ts`. `background.ts` owns native messaging to the native host/supervisor.
- DISCOVERY (first step): find the existing popup→background→native-host message path in `background.ts` and reuse it for RPC passthrough. Do NOT open a second native-messaging port from the popup.
- Extension pages CSP is `script-src 'self'` — no inline scripts.
- Supervisor methods available: `status.snapshot`, `automation.pause/resume`, `runs.cancel`, `runs.list/get`, plus the rest from EUI-T-0001..0003.
- Design intent: one UI with progressive disclosure — fleet features appear only when enrolled; there is no separate "fleet mode".

## Contract

- `extension/src/ui/rpc.ts`: `rpc<T>(method, params)` → `chrome.runtime.sendMessage` → background → supervisor. Per-method zod response schemas, timeout, and a typed `SupervisorUnreachable` error every screen can render.
- Popup rebuild (`popup.ts`/`popup.css`, target ~420×600):
  - Health strip: extension ↔ native host ↔ supervisor ↔ cloud (last-poll age when enrolled), traffic-light dots.
  - Now-running card: workflow, origin badge (local/fleet/schedule), step N of M, elapsed, Stop button → `runs.cancel`.
  - Master "Pause automation" toggle → `automation.pause/resume` (confirm dialog when a run is in flight, offering cancel_current).
  - Recent 5 runs with status dots; click deep-links to `dashboard.html#runs/<run_id>`.
  - Footer: enrolled → device name/tenant/status; not enrolled → "Connect to a server" link → `#fleet`.
  - Refresh `status.snapshot` every 2s while open; render skeleton + reconnect hint on `SupervisorUnreachable`.
- Badge (in `background.ts`, logic as a pure tested fn `state -> {text, color}`): running count (blue), red `!` when last run failed or `flagged_workflows > 0`, `⏸` when paused, gray dot when supervisor unreachable. Update via a `chrome.alarms` 30s tick plus piggyback on popup polls.
- Dashboard scaffold: `extension/dashboard.html`, `extension/src/dashboard/` with hash routing (`#runs`, `#workflows`, `#fleet`, `#logs`, `#settings`), shared nav/layout, shared rpc client, empty tab stubs. New `vite.config.dashboard.ts` mirroring the popup config; wire into `build.sh` / package scripts so both chrome and firefox dists include it.

## File ownership

- OWN: `extension/src/popup.ts`, `extension/src/popup.css`, `extension/popup.html`, `extension/src/ui/**` (new), `extension/src/dashboard/**` (new), `extension/dashboard.html` (new), `extension/vite.config.dashboard.ts` (new), build-script registration lines.
- SHARED (additive, small): `background.ts` — one message-router case for supervisor RPC passthrough + the badge updater.
- OFF-LIMITS: `contentScript.ts`, `pageBridge.ts`, `crates/**`.

## Constraints

- No new runtime framework dependency (React/Vue/etc.) without epic-owner sign-off — vanilla TS + small render helpers, matching the repo.
- Every screen handles `SupervisorUnreachable` gracefully.
- vitest for unit tests, mocking chrome APIs the way existing extension tests do.
- No AI attribution anywhere.

## Dependencies

- EUI-T-0003 (`status.snapshot`, pause, cancel). EUI-T-0001 for run summaries.
