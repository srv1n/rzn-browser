# SEC-T-0003 · Remove chat_json envelope-as-result fallback in rzn_plan llm.rs

## Intent

`chat_json` in `rzn_plan` has a last-resort fallback that returns the entire raw LLM API envelope as the "parsed" JSON result whenever real parsing fails. Downstream planners then consume garbage as if it were a plan. Remove the fallback so parse failures surface as errors.

## Context (read before coding)

Audit anchor (2026-06-11, re-verify): `crates/rzn_plan/src/llm.rs:1141-1148`. The function stringifies the full Responses API envelope (`response.to_string()`) and then scans `find('{')` / `rfind('}')`. Because the envelope itself is a JSON object, this **always matches the whole envelope**, so stage 3 cannot fail — every malformed model output is silently replaced by the raw API response (ids, usage metadata, content blocks) and handed to callers expecting a plan/step object.

Read the whole `chat_json` function first and map its stages: typically (1) parse extracted `output_text` directly, (2) extract from fenced ```json blocks, (3) the broken brace-scan. Identify every caller (`grep -rn "chat_json" crates/rzn_plan/src`) and confirm they handle an `Err` (they already must, since stages 1-2 can error before stage 3 was added).

## Implementation plan

1. Delete the stage-3 brace-scan over the envelope. If a brace-scan heuristic is worth keeping at all, run it **only over the extracted assistant text**, never over the envelope string.
2. Return a structured error that includes a bounded snippet (first ~500 chars) of the assistant text that failed to parse — not the full envelope, which can embed page data.
3. Unit tests: (a) valid JSON in output_text → parsed; (b) JSON inside a fenced block → parsed; (c) prose/no-JSON output → `Err`, and the error does NOT contain envelope keys like `"usage"`/`"id"`; (d) truncated JSON → `Err`.
4. Check sibling helpers in the same file for the same brace-scan pattern (`grep -n "rfind('}')" crates/rzn_plan/src`) and fix any copies.
