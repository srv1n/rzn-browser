# EUI-T-0002 ¬∑ Failure fingerprinting and workflow health flags

## Intent

Turn repeated workflow failures into a first-class signal instead of a log line: classify failures, fingerprint them, compute rolling per-workflow health with degraded/broken flags, capture bounded failure context, and expose it all over RPC. The repeating-fingerprint case ("failed 4√ó at step 6, selector_not_found, since Jul 9") is the actionable output ‚Äî it means the site changed, not random flake.

## Context

- `crates/rzn_browser/src/workflow_failure_report.rs` already builds failure context (`build_failure_context`, `WorkflowRunFailure`) ‚Äî reuse/extend its error analysis; do not build a parallel system.
- Run history comes from the run store (EUI-T-0001).
- CROSS-REPO CANON (backend FLD-T-0001 consumes this ‚Äî never change one side alone): `RunResultV2` gains an optional `failure_summary` object `{error_class, failing_step_index, fingerprint, message}`. `error_class` is one of exactly: `selector_not_found | timeout | navigation | auth_wall | popup_blocked | engine_error | cancelled | unknown`. `fingerprint` is PINNED byte-for-byte (2026-07-12, matches the landed `workflow_health.rs` impl): `hex(sha256("{workflow_hash}:{failing_step_index}:{error_class}"))[..16]`, where a missing step index serializes as the empty string (i.e. `"{hash}::{class}"`). Backend FLD-T-0001 must reproduce this exactly.

## Contract

New module `crates/rzn_browser/src/workflow_health.rs`:
- `classify_error(...) -> ErrorClass` mapping engine/step errors onto the class enum (unknown as fallback, never panic).
- `fingerprint(record) -> String` per the canon above.
- `compute_health(records) -> WorkflowHealth { flag, consecutive_failures, success_rate_last_20, dominant_fingerprint: Option<{step_index, error_class, count, first_seen_at}> }` ‚Äî flags: `healthy`; `degraded` at ‚â•3 consecutive failures; `broken` when the same fingerprint occurs ‚â•3 times in a row. Thresholds are consts overridable via `RZN_HEALTH_DEGRADED_AFTER` / `RZN_HEALTH_BROKEN_AFTER`. Pure functions over synthetic records ‚Äî no clock, no sleeps in tests.
- RPC arm `workflows.health {} -> {workflows: [{workflow_id, workflow_hash, last_run_at, health}]}` derived from the run store.
- Contracts change: add `failure_summary: Option<FailureSummaryV1>` to `RunResultV2` in `rzn_contracts` (serde default, additive ‚Äî old JSON without the field must still deserialize). Populate it on failed runs in the runner completion path so fleet result posts carry it to the backend.
- Failure context capture: on a failed run, persist `<base>/runs/<run_id>.failure.json` with console tail (bounded ‚â§4KB) and, when the transport can supply one, a screenshot (base64) or DOM excerpt of the failing step. Capture failure must NEVER fail the run ‚Äî record `capture_unavailable: <reason>` instead.

## File ownership

- OWN: `crates/rzn_browser/src/workflow_health.rs` (new), `FailureSummaryV1` in `rzn_contracts`, failure-capture hookup in the runner completion path, `workflows.health` RPC arm in `supervisor.rs`.
- SHARED (append-only): `run_store.rs` record fields (`error_class`, fingerprint).
- OFF-LIMITS: `extension/**`, `workflow_failure_report.rs` semantics (extend, don't rewrite).

## Constraints

- Health computation must be pure and deterministic (records in, health out).
- Bounded capture sizes; screenshots only on failure, never on success.
- No AI attribution anywhere.

## Dependencies

- EUI-T-0001 (run store).
