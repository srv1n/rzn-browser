# EUI-T-0009 · Cloud-job notifications and background-window isolation

## Intent

Device-owner courtesy for cloud-dispatched work: a Chrome notification when a fleet job starts or fails, and fleet runs isolated into a separate unfocused Chrome window so remote work never steals the human's active tab. Small task, outsized perceived-quality impact — and the core trust feature alongside the pause button.

## Context

- The manifest already has the `notifications` permission.
- The supervisor already pushes commands to the extension (`dispatch_cloud_command_to_extension` in `supervisor.rs`); reuse that channel for a new event kind rather than inventing a new one.
- The fleet loop knows run origin; notification settings (`notifications_enabled`, `notify_on`, `fleet_keep_window_on_failure`) come from EUI-T-0008 — the supervisor checks settings BEFORE emitting the notice, so the extension side stays dumb.
- WINDOW-STATE TRAP: minimized windows get occlusion-throttled by Chrome (timers/rendering), which can break real workflows. Default must be a NORMAL, unfocused window (`chrome.windows.create {focused: false}`), NOT minimized.

## Contract

- Supervisor: emit `fleet_run_notice {job_id, workflow_id, phase: "started"|"succeeded"|"failed", error_class?}` events over the existing command channel at fleet-run start/end, gated by settings.
- Extension (`background.ts` + a small owned module): on notice, `chrome.notifications.create` — start: "Fleet job: <workflow_id> started"; failure: "<workflow_id> failed (<error_class>)". Clicking the notification opens `dashboard.html#runs` (deep link to the run when the id is known).
- Window isolation: when a run's origin is fleet, the run's first tab-open creates a dedicated unfocused window; subsequent tab operations for that run target that window; on run end the window closes — unless the run failed and `fleet_keep_window_on_failure` is true (leave it open for debugging).
- DISCOVERY (first step): find where the extension currently picks the target window/tab for open-tab commands and scope the change to window selection only — do NOT touch step execution semantics. Plumb a per-run `window_id` through whatever run/session context already exists on the extension side.
- Local runs (CLI/MCP) keep today's behavior exactly.

### Contract amendment (2026-07-12) — session-lifecycle protocol fields AUTHORIZED

First implementation attempt confirmed `fleet_run_notice` cannot key window isolation (it is notification-settings-gated and carries no session id); inferring "next tab command belongs to the fleet run" would corrupt concurrent local runs. Isolation therefore rides the session lifecycle messages. The `crates/**` restriction is LIFTED for exactly this additive change (see `tusker/feedback/agents/EUI-T-0009-window-isolation-contract-gap.md`):

- `browser.session_open` payload gains optional fields: `origin` (`"local_cli" | "mcp" | "fleet"`), `run_id`, `job_id?` — set wherever the supervisor/runner constructs session_open (the fleet loop knows origin and job id; the runner knows run_id).
- `browser.session_close` payload gains optional fields: `outcome` (`"succeeded" | "failed" | "cancelled" | "timed_out"`) and `keep_window: bool`. The SUPERVISOR resolves `keep_window` (= run failed AND `fleet_keep_window_on_failure` setting) so the extension stays dumb — same doctrine as notice gating.
- All fields optional/additive with serde defaults; messages without them behave exactly as today, so local runs cannot regress.
- Extension: key the dedicated unfocused window by `run_id` (fallback `session_id`); only `origin == "fleet"` sessions get isolation; `session_close` applies `keep_window`.
- `fleet_run_notice` stays purely a notification concern — no coupling to isolation.
- Supervisor-side test required: fleet-origin session_open carries origin/run_id/job_id and session_close carries resolved outcome/keep_window; local-origin sessions omit fleet fields and never trigger isolation.

## File ownership

- OWN: notification module `extension/src/ui/notifications.ts` (new), window-isolation change at the discovered window-selection point, `fleet_run_notice` emission in `supervisor_fleet.rs`, and (per the 2026-07-12 amendment) the additive session_open/session_close payload fields at their supervisor/runner construction sites plus the settings-resolved `keep_window` logic.
- SHARED (additive, small): `background.ts` router case.
- OFF-LIMITS: `contentScript.ts`, `pageBridge.ts`, step execution semantics in `crates/**` (payload fields are authorized; changing step/transport behavior is not).

## Constraints

- Never notify for local runs. Never focus/raise the fleet window programmatically.
- Window cleanup must be best-effort safe (window already closed by the user → no error surfaced to the run result).
- No AI attribution anywhere.
