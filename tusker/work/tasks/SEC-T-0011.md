# SEC-T-0011 · Reconcile CDP debugger leases across MV3 service-worker restarts

## Intent

The CDP debugger detach lifecycle depends on a `setInterval` lease poller inside the MV3 service worker. When Chrome suspends or kills the service worker mid-lease, the timer dies and `chrome.debugger` stays attached (the "started debugging this browser" banner persists) until the tab closes. Make lease expiry survive service-worker restarts and reconcile orphaned attachments on startup.

## Context (read before coding)

Audit anchors (2026-06-11, re-verify):

1. `extension/src/runtime/cdp_session_manager.ts:66` — per-action `handle.release()` is a deliberate no-op (comment explains leases handle detach).
2. `extension/src/background.ts:31-292` — CDP lease manager + circuit breaker; the lease expiry poller is a `setInterval` around `:39`. MV3 service workers are suspended after ~30s idle and `setInterval` does not survive suspension; `chrome.alarms` does (minimum period 30s, and `chrome.alarms.create` with `when` supports sub-minute one-shots from MV3 Chrome 120+ — verify the minimum the target Chrome accepts and design around a 30s granularity floor).
3. There is no startup reconciliation: after SW restart, previously attached targets are unknown to the new SW instance.

Design constraints: CDP is off-by-default with per-domain flags and TTL leases (keep that model). Detach must never fire while a lease is legitimately active — drive expiry from persisted lease state, not in-memory timers alone.

## Implementation plan

1. **Persist lease state**: store active leases (`tabId`, `targetId`, `expiresAt`, purpose) in `chrome.storage.session` (survives SW restarts within the browser session, cleared on browser exit — exactly the lifetime we want).
2. **Alarm-driven expiry**: replace/augment the `setInterval` with a `chrome.alarms` alarm ("cdp-lease-sweep", periodInMinutes: 0.5 if accepted, else 1). Keep the in-memory interval as a fast path while the SW is alive; the alarm is the durability backstop. On alarm: load persisted leases, detach expired ones, clean storage.
3. **Startup reconciliation**: on SW init (top-level of background script + `chrome.runtime.onStartup`), call `chrome.debugger.getTargets()`, find targets with `attached === true` that belong to us (match against persisted leases; targets attached by us with no surviving lease record are orphans), and detach orphans. Wrap in try/catch — `getTargets` requires the `debugger` permission which the manifest already has.
4. **Detach path**: ensure `chrome.debugger.onDetach` (user-cancelled banner) clears the persisted lease so reconciliation doesn't re-detach a dead handle.
5. **Tests**: vitest unit tests for the lease store (persist/expire/reconcile decision logic) with mocked `chrome.*` APIs (follow the existing mocking pattern in `extension/src/*.test.ts`). E2E if feasible: the Playwright CDP suites (`.pw-user-data-cdp-queue` projects exist) should still pass; add an assertion that after a forced SW restart (`chrome://serviceworker-internals` is not scriptable — instead use the `chrome.runtime.reload()` route or skip e2e and rely on unit-level decision tests).
